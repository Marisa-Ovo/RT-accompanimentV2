# 实时钢琴伴奏生成系统：方法论分析

## 面向论文写作与深度理解的技术文档

---

## 一、问题定义与研究动机

### 1.1 任务定义

**实时音乐伴奏生成**：给定一个实时输入的旋律序列 $X = \{x_1, x_2, ..., x_T\}$（Part0），系统需要生成与之匹配的伴奏序列 $Y = \{y_1, y_2, ..., y_T\}$（Part1），且满足**低延迟约束**——即在接收到 $x_t$ 后，需要在有限时间内生成 $y_t$。

### 1.2 现有方法的局限性

| 方法类别 | 代表工作 | 局限性 |
|----------|----------|--------|
| **Seq2Seq模型** | Music Transformer | 需要完整输入序列，无法实时生成 |
| **帧级生成** | WaveNet系列 | 计算开销大，难以保持长程结构 |
| **小节级生成** | MuseNet | 粒度过粗，响应延迟高 |
| **独立生成** | MIDI-VAE | 缺乏条件控制，伴奏与旋律不协调 |

### 1.3 核心挑战

1. **因果性约束**：生成 $y_t$ 时，理论上只能观测到 $x_{1:t}$，但实际音乐中伴奏与旋律存在复杂的时序依赖
2. **实时性要求**：生成延迟需控制在音乐可感知阈值内（通常 < 100ms）
3. **音乐连贯性**：生成的伴奏需保持和声、节奏的长程一致性
4. **条件控制**：伴奏需与旋律在调性、节奏、风格上保持协调

---

## 二、方法概述

### 2.1 核心思想

本方法提出**Beat级交错生成框架**，将传统的序列到序列问题转化为**交错的条件语言建模问题**：

$$P(Y|X) = \prod_{t=1}^{T} P(y_t | x_{1:t+\delta}, y_{1:t-1})$$

其中 $\delta$ 为时间偏移参数（`delay_beats`），控制伴奏相对旋律的提前或延迟量。

### 2.2 方法框架图

```
┌────────────────────────────────────────────────────────────────────┐
│                        训练阶段                                     │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  输入序列: [BOS][TS][BPM][P0_b0][P1_b0][P0_b1][P1_b1]...[EOS]      │
│                          ↓                                         │
│  标签序列: [PAD][PAD][PAD][PAD ][P1_b0][PAD ][P1_b1]...            │
│                                   ↑                                │
│                          仅对Part1计算损失                          │
│                                                                    │
├────────────────────────────────────────────────────────────────────┤
│                        推理阶段                                     │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  Step 1: 注入 Part0_beat_0  →  清空KV缓存                          │
│  Step 2: 自回归生成 Part1_beat_0                                   │
│  Step 3: 注入 Part0_beat_1  →  清空KV缓存                          │
│  Step 4: 自回归生成 Part1_beat_1                                   │
│  ...                                                               │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

## 三、关键技术设计

### 3.1 Beat级交错表示

#### 3.1.1 设计动机

音乐具有天然的层次结构：音符 → 拍 → 小节 → 乐句。选择**拍（Beat）**作为交错粒度的原因：

| 粒度 | 优势 | 劣势 |
|------|------|------|
| **音符级** | 最细粒度控制 | 序列过长，计算开销大 |
| **帧级** | 适合音频生成 | 符号音乐中冗余度高 |
| **拍级** | 符合音乐节奏结构，序列长度适中 | 需要对齐处理 |
| **小节级** | 序列最短 | 粒度过粗，实时性差 |

#### 3.1.2 形式化定义

设一个音乐片段包含 $M$ 个小节，每小节 $B$ 拍，则：
- Part0序列：$X = \{x_1^{(0)}, x_2^{(0)}, ..., x_{M \times B}^{(0)}\}$
- Part1序列：$Y = \{y_1^{(1)}, y_2^{(1)}, ..., y_{M \times B}^{(1)}\}$

交错后的训练序列：
$$S = [BOS, TS, BPM, x_1^{(0)}, y_1^{(1)}, x_2^{(0)}, y_2^{(1)}, ..., x_n^{(0)}, y_n^{(1)}, EOS]$$

### 3.2 时间偏移机制（delay_beats）

#### 3.2.1 问题背景

在真实的实时伴奏场景中，存在一个根本性矛盾：
- **理想情况**：伴奏 $y_t$ 应与旋律 $x_t$ 同步
- **因果约束**：生成 $y_t$ 时必须已知条件 $x_t$

#### 3.2.2 解决方案

引入时间偏移参数 $\delta$（`delay_beats`）：

**情况1：$\delta = 0$（同步模式）**
```
时间轴:  t=0    t=1    t=2    t=3
Part0:   x_0    x_1    x_2    x_3
Part1:   y_0    y_1    y_2    y_3
生成y_t时可见: x_{0:t}
```

**情况2：$\delta > 0$（延迟模式）**
```
时间轴:  t=0    t=1    t=2    t=3
Part0:   x_0    x_1    x_2    x_3
Part1:   [PAD]  y_0    y_1    y_2
生成y_t时可见: x_{0:t+δ}（更多未来信息）
```

**情况3：$\delta < 0$（提前模式，实时伴奏）**
```
时间轴:  t=0    t=1    t=2    t=3
Part0:   [PAD]  x_0    x_1    x_2
Part1:   y_0    y_1    y_2    y_3
生成y_t时可见: x_{0:t+δ}（更少信息，需要预测）
```

#### 3.2.3 实现方式

通过在序列中插入填充标记实现时间偏移：

```python
if delta >= 0:  # Part1延迟
    Part0_padded = Part0 + [PAD] * delta
    Part1_padded = [PAD] * delta + Part1
else:           # Part1提前（实时模式）
    Part0_padded = [PAD] * |delta| + Part0
    Part1_padded = Part1 + [PAD] * |delta|
```

#### 3.2.4 理论分析

当 $\delta = -1$ 时，模型需要学习：
$$P(y_t | x_{1:t-1}, y_{1:t-1})$$

即在**未观测到对应旋律**的情况下预测伴奏。这要求模型学习：
1. 旋律的局部模式和走向预测
2. 伴奏的内在连贯性
3. 旋律-伴奏的统计相关性

### 3.3 稀疏相对位置编码

#### 3.3.1 动机

钢琴卷帘（Piano Roll）是一个稀疏矩阵：
- 维度：88（音高）× T（时间步）
- 稀疏度：通常 > 95%（大部分位置为静音）

直接展平会导致序列过长且充满冗余的零值。

#### 3.3.2 编码方案

**Step 1: Patch化三进制编码**

将 $1 \times 4$ 的时间窗口编码为单个token：

| 状态组合 | sustain | onset | 编码值 |
|----------|---------|-------|--------|
| 静音 | 0 | 0 | 0 |
| 延音 | 1 | 0 | 1 |
| 起音 | 1 | 1 | 2 |

对于 $1 \times 4$ patch：$token = \sum_{i=0}^{3} v_i \cdot 3^{3-i}$，范围 $[0, 80]$

**Step 2: 相对位置压缩**

只记录非零位置，使用相对距离编码：

```
原始序列: [0, 0, 50, 0, 0, 60, 0, 40, ...]  (88维)
          位置: 0  1   2  3  4   5  6   7

压缩后:   [pos_marker+2, 50, pos_marker+3, 60, pos_marker+2, 40, end_marker]
          (相对位置2)      (相对位置3)      (相对位置2)
```

#### 3.3.3 压缩效率分析

设原始序列长度为 $L$，非零元素数量为 $K$，则：
- 原始表示：$O(L)$
- 压缩表示：$O(2K + 1)$（每个非零元素需要位置标记+值+结束标记）

对于稀疏度 $s = 1 - K/L$ 的序列，压缩比为：
$$r = \frac{2K + 1}{L} = \frac{2(1-s)L + 1}{L} \approx 2(1-s)$$

当 $s = 95\%$ 时，$r \approx 0.1$，即压缩到原来的 10%。

### 3.4 选择性损失计算

#### 3.4.1 设计原理

训练目标是学习条件分布 $P(Y|X)$，而非联合分布 $P(X, Y)$。因此：
- Part0（旋律）：作为**条件输入**，不需要预测
- Part1（伴奏）：作为**生成目标**，计算损失

#### 3.4.2 实现方式

```python
# 构造标签序列
for p0, p1 in zip(part0_beats, part1_beats):
    input_tokens.extend([p0, p1])           # 输入包含两部分
    label_tokens.extend([PAD_TOKEN, p1])    # 标签中Part0被mask
```

损失函数：
$$\mathcal{L} = -\sum_{t} \mathbb{1}[label_t \neq PAD] \cdot \log P(label_t | input_{1:t})$$

#### 3.4.3 优势分析

| 方面 | 联合建模 $P(X,Y)$ | 条件建模 $P(Y|X)$ |
|------|-------------------|-------------------|
| 模型容量 | 分散到X和Y | 集中到Y |
| 训练效率 | 较低 | 较高 |
| 生成质量 | 可能复制X | 专注于Y的生成 |
| 推理灵活性 | 可生成X和Y | 只能生成Y |

## 四、模型架构

### 4.1 整体架构

基于LLaMA的因果语言模型：

```
输入嵌入 (Embedding)
    ↓
RoPE位置编码 (Rotary Position Embedding)
    ↓
┌─────────────────────────┐
│   Transformer Block ×18 │
│   ├─ RMSNorm            │
│   ├─ Causal Self-Attn   │
│   ├─ RMSNorm            │
│   └─ SwiGLU FFN         │
└─────────────────────────┘
    ↓
RMSNorm
    ↓
LM Head (线性投影到词表)
```

### 4.2 关键超参数

| 参数 | 值 | 设计考量 |
|------|-----|----------|
| 隐层维度 | 768 | 平衡容量与效率 |
| 注意力头数 | 6 | 每头128维，标准配置 |
| 层数 | 18 | 足够的表示能力 |
| FFN维度 | 3072 | 4倍隐层维度 |
| 最大序列长度 | 3500 | 覆盖大部分音乐片段 |
| 词表大小 | 268 | 紧凑的token空间 |
| RoPE theta | 10000 | 标准位置编码配置 |

### 4.3 为什么选择LLaMA架构

| 特性 | 优势 |
|------|------|
| **RMSNorm** | 比LayerNorm更稳定，计算更快 |
| **SwiGLU** | 比ReLU/GELU更好的表达能力 |
| **RoPE** | 相对位置编码，更好的长度外推 |
| **因果注意力** | 天然支持自回归生成 |

### 5.2 长度感知批处理

**问题**：音乐序列长度差异大，直接padding导致计算浪费

**解决方案**：Bucket Batch Sampler

```
Step 1: 按序列长度排序所有样本
Step 2: 分成固定大小的bucket（如100个样本）
Step 3: 训练时随机打乱bucket顺序
Step 4: bucket内部随机采样形成batch
```

**效果**：同一batch内序列长度相近，减少padding开销约40-60%

### 5.3 数据增强

**音高平移**（70%概率）：
$$x'_{pitch} = x_{pitch} + \Delta, \quad \Delta \sim \text{Uniform}(-5, 5)$$

**作用**：
1. 增加数据多样性
2. 使模型学习调性无关的模式
3. 防止过拟合到特定调性

## 六、推理机制

### 6.1 KV缓存管理

**标准KV缓存**：
- 缓存历史token的Key和Value
- 新token只需计算与缓存的注意力
- 复杂度从 $O(n^2)$ 降到 $O(n)$

**本方法的特殊处理**：
- 手动注入Part0 token时，缓存失效
- 必须重置缓存，重新计算注意力
- 每个beat需要一次缓存重置

### 6.2 采样策略

```python
def sample(logits, temperature, top_k, top_p, rep_penalty):
    # 1. 温度调节
    logits = logits / temperature

    # 2. 重复惩罚
    for token in generated:
        logits[token] /= rep_penalty

    # 3. Top-K截断
    if top_k > 0:
        threshold = topk(logits, k=top_k)[-1]
        logits[logits < threshold] = -inf

    # 4. Top-P (Nucleus) 截断
    if top_p < 1.0:
        sorted_probs = cumsum(softmax(sort(logits)))
        cutoff_idx = searchsorted(sorted_probs, top_p)
        logits[sorted_idx > cutoff_idx] = -inf

    # 5. 采样
    return multinomial(softmax(logits))
```

### 6.3 GT前缀注入

**机制**：生成的前 $N$ 拍使用Ground Truth

**作用**：
1. 提供生成起点，避免冷启动问题
2. 建立正确的和声/节奏上下文
3. 类似于Prompt的作用

**参数选择**：$N = 12$（约3个小节）通常效果较好

## 十、局限性与未来工作

### 10.1 当前局限性

1. **固定delay_beats**：训练和推理时delay_beats需一致，缺乏灵活性
2. **KV缓存效率**：频繁的缓存重置降低推理效率
3. **音乐表示**：仅支持钢琴，不支持多音色
4. **长程依赖**：超长序列（>3500 tokens）会被截断
5. **评估困难**：缺乏标准化的实时伴奏评估基准

### 10.2 未来改进方向

1. **动态delay_beats**：训练时随机化delay_beats，推理时可调
2. **Speculative Decoding**：并行预测多个token，提高效率
3. **多轨扩展**：支持鼓、贝斯等多种乐器
4. **流式KV缓存**：设计适合交错生成的缓存机制
5. **在线学习**：根据用户反馈实时调整生成策略

## 十一、核心贡献总结

本方法的主要贡献可归纳为以下四点：

### 贡献1：Beat级交错生成框架
提出将实时伴奏生成建模为beat级交错的条件语言建模问题，实现了streaming式生成能力。

### 贡献2：灵活的时间偏移机制
通过delay_beats参数统一建模提前、同步、延迟三种伴奏模式，为不同应用场景提供灵活支持。

### 贡献3：高效的稀疏位置编码
设计相对位置压缩编码，将序列长度压缩60-80%，显著提高训练和推理效率。

### 贡献4：选择性损失计算
通过仅对伴奏部分计算损失，使模型专注于条件生成任务，提高生成质量。

## 十三、损失函数详解

### 13.1 损失函数的本质

本系统使用**标准的因果语言模型损失（Causal LM Loss）**，即交叉熵损失，但通过**标签掩码**实现选择性计算：

```python
loss = CrossEntropyLoss(ignore_index=pad_token_id)(logits, labels)
```

### 13.2 标签构造机制

关键在于标签序列的构造方式（`PianoDataset.py:332-339`）：

```python
for p0, p1 in zip(part0_padded, part1_padded):
    # 输入序列：Part0和Part1都包含
    all_input_tokens.append(p0)
    all_input_tokens.append(p1)

    # 标签序列：Part0位置用pad_token替换（不计算loss）
    all_label_tokens.append(torch.full_like(p0, self.pad_token))  # ← 关键
    all_label_tokens.append(p1)
```

### 13.3 数据流示意

```
位置索引:     0     1     2      3      4      5      6      7     ...
            [BOS] [TS] [BPM] [P0_b0] [P1_b0] [P0_b1] [P1_b1] [P0_b2] ...

input_ids:  [257] [259] [264]  [...]   [...]   [...]   [...]   [...]
labels:     [PAD] [PAD] [PAD]  [PAD]   [...]   [PAD]   [...]   [PAD]
                                ↑               ↑               ↑
                           Part0位置         Part0位置       Part0位置
                           不计算loss        不计算loss      不计算loss
```

### 13.4 损失计算过程

在HuggingFace的`LlamaForCausalLM`内部：

```python
# 1. 模型前向传播，得到每个位置的logits
logits = model(input_ids)  # shape: (batch, seq_len, vocab_size)

# 2. Shift操作：预测下一个token
shift_logits = logits[..., :-1, :]      # 去掉最后一个位置
shift_labels = labels[..., 1:]          # 去掉第一个位置

# 3. 计算交叉熵，自动忽略label为pad_token的位置
loss = CrossEntropyLoss(ignore_index=pad_token_id)(
    shift_logits.view(-1, vocab_size),
    shift_labels.view(-1)
)
```

### 13.5 具体示例

假设序列为：`[BOS, TS, BPM, P0_b0, P1_b0, P0_b1, P1_b1]`

| 位置 | 预测目标 | 标签 | 是否计算loss |
|------|----------|------|--------------|
| BOS → | TS | PAD | ❌ |
| TS → | BPM | PAD | ❌ |
| BPM → | P0_b0 | PAD | ❌ |
| P0_b0 → | P1_b0 | **P1_b0** | ✅ |
| P1_b0 → | P0_b1 | PAD | ❌ |
| P0_b1 → | P1_b1 | **P1_b1** | ✅ |

**结果**：模型只学习"给定Part0后预测Part1"，不学习预测Part0本身。

### 13.6 数学形式

设序列长度为 $N$，Part1位置集合为 $\mathcal{P}_1$，则损失函数为：

$$\mathcal{L} = -\frac{1}{|\mathcal{P}_1|} \sum_{t \in \mathcal{P}_1} \log P_\theta(x_t | x_{<t})$$

其中 $x_t$ 表示位置 $t$ 的token，$x_{<t}$ 表示位置 $t$ 之前的所有token（包含Part0和之前的Part1）。

## 十四、为什么其他模型架构不能实现交错生成

### 14.1 核心前提：因果语言模型的自回归特性

本方法的交错设计**依赖于因果语言模型的特性**：

```
因果注意力掩码：位置i只能看到位置[0, 1, ..., i-1]的信息

     BOS  TS  BPM  P0_0  P1_0  P0_1  P1_1
BOS   ✓   ✗   ✗    ✗     ✗     ✗     ✗
TS    ✓   ✓   ✗    ✗     ✗     ✗     ✗
BPM   ✓   ✓   ✓    ✗     ✗     ✗     ✗
P0_0  ✓   ✓   ✓    ✓     ✗     ✗     ✗
P1_0  ✓   ✓   ✓    ✓     ✓     ✗     ✗   ← 生成P1_0时能看到P0_0
P0_1  ✓   ✓   ✓    ✓     ✓     ✓     ✗
P1_1  ✓   ✓   ✓    ✓     ✓     ✓     ✓   ← 生成P1_1时能看到P0_0, P1_0, P0_1
```

### 14.2 其他架构的局限性

#### 14.2.1 Encoder-Decoder架构（原版Transformer、T5等）

```
Encoder: 处理完整的Part0序列
         ↓
Decoder: 生成Part1序列
```

**无法交错的原因**：
- Encoder需要**完整的Part0**才能开始解码
- 无法实现"边接收Part0边生成Part1"
- 本质上是**离线**的，不支持流式处理

#### 14.2.2 双向模型（BERT、双向Attention）

```
双向注意力：每个位置能看到所有其他位置
```

**无法交错的原因**：
- 生成P1_0时能看到P0_1、P0_2...（未来信息泄露）
- 训练和推理不一致：训练时有未来信息，推理时没有
- 无法用于自回归生成

#### 14.2.3 传统RNN/LSTM

```
h_t = RNN(x_t, h_{t-1})
```

**理论上可行，但实际问题**：
- 难以并行训练（必须顺序处理）
- 长程依赖能力弱
- 没有KV缓存机制，推理效率低
- 现代实践中已被Transformer取代

#### 14.2.4 Diffusion Models

```
从噪声逐步去噪生成完整序列
```

**无法交错的原因**：
- 生成过程是**全局**的，一次性生成整个序列
- 难以实现逐token/逐beat的流式生成
- 条件注入方式不同（通常是cross-attention）

### 14.3 因果LM可行的原因

| 特性 | 因果LM | 作用 |
|------|--------|------|
| **因果注意力** | 位置i只看[0:i-1] | 保证训练推理一致性 |
| **自回归生成** | 逐token生成 | 支持流式输出 |
| **KV缓存** | 缓存历史计算 | 高效增量推理 |
| **统一序列建模** | 输入输出同一序列 | 灵活的条件注入 |

### 14.4 架构对比总结

| 架构 | 能否交错 | 原因 |
|------|----------|------|
| **Encoder-Decoder** | ❌ | 需要完整输入才能解码 |
| **双向Transformer** | ❌ | 训练时有未来信息泄露 |
| **Diffusion** | ❌ | 全局去噪，非流式生成 |
| **RNN/LSTM** | △ | 理论可行但效率低、能力弱 |
| **因果LM（本方法）** | ✅ | 因果注意力+自回归+KV缓存 |

## 十五、为什么其他音乐编码方式不能实现交错生成

### 15.1 核心问题：编码方式决定了"声部边界"是否清晰

不同编码方式对多声部音乐的处理方式决定了能否实现交错生成。

### 15.2 MIDI事件序列编码（Music Transformer、MuseNet等）

**编码方式**：
```
[note_on_60] [time_shift_120] [note_off_60] [note_on_72] [time_shift_60] ...
```

**多声部处理**：
```
时间轴:     t=0         t=120        t=180
Part0:    note_on_72              note_off_72
Part1:    note_on_48   note_on_52  note_off_48

混合后的事件序列（按时间排序）:
[note_on_72] [note_on_48] [time_shift_120] [note_on_52] [time_shift_60] ...
     ↑            ↑                              ↑
   Part0?      Part1?                         Part1?
```

**无法交错的原因**：
- 事件按**时间顺序**排列，不是按**声部**排列
- 同一时刻的Part0和Part1事件**混在一起**
- 模型无法知道"当前应该生成Part0还是Part1"
- 没有明确的**声部边界标记**

### 15.3 REMI编码

**编码方式**：
```
[Bar] [Position_0] [Note_on_60] [Duration_2] [Position_2] [Note_on_64] ...
```

**无法交错的原因**：
- 同样是时间优先，不是声部优先
- Position token标记的是**绝对时间位置**
- 多声部音符在同一Position下混合
- 难以区分"这个音符属于Part0还是Part1"

### 15.4 Compound Word / CP编码

**编码方式**：
```
每个音符 = [Type, Bar, Position, Pitch, Duration, Velocity]
```

**无法交错的原因**：
- 虽然可以加入Track/Part字段，但生成时：
  - 需要先决定Type，再决定其他属性
  - "该生成Part0还是Part1"变成了**预测问题**而非**结构问题**
  - 模型可能在错误的时机生成错误的声部

### 15.5 本方法编码的关键设计

**序列结构**：
```
[BOS][TS][BPM][P0_beat_0][P1_beat_0][P0_beat_1][P1_beat_1]...
              ↑          ↑          ↑          ↑
           位置固定    位置固定    位置固定    位置固定
           一定是P0    一定是P1    一定是P0    一定是P1
```

**核心优势**：

| 特性 | 作用 |
|------|------|
| **Beat级分割** | 固定的时间粒度，提供同步点 |
| **位置决定声部** | 奇数位置=Part0，偶数位置=Part1 |
| **独立的end_marker** | Part0用170，Part1用171，边界清晰 |
| **相对位置编码** | 每个beat内部独立编码，不跨beat依赖 |

**生成时模型"知道"该做什么**：
```
当前序列: [BOS][TS][BPM][P0_b0][P1_b0][P0_b1]
                                        ↑
                                   刚注入Part0

下一个位置一定是Part1_beat_1，模型只需要：
1. 看到P0_b0, P1_b0, P0_b1作为上下文
2. 生成Part1_beat_1的token序列
3. 直到遇到end_marker(171)表示这个beat结束
```

### 15.6 编码方式对比总结

| 编码方式 | 声部边界 | 时间对齐 | 能否交错生成 |
|----------|----------|----------|--------------|
| MIDI事件序列 | ❌ 混合 | ❌ 连续 | ❌ |
| REMI | ❌ 混合 | △ Position | ❌ |
| Compound Word | △ 可加字段 | △ Position | △ 困难 |
| Piano Roll展平 | ✅ 可分离 | ✅ 离散 | △ 序列太长 |
| **本方法** | ✅ 位置固定 | ✅ Beat对齐 | ✅ |

### 15.7 本质差异

```
其他编码方式：时间优先
  序列 = [t0的所有音符] [t1的所有音符] [t2的所有音符] ...
  → 声部信息是附属属性，边界模糊

本方法：声部优先（通过位置交错）
  序列 = [P0_beat0] [P1_beat0] [P0_beat1] [P1_beat1] ...
  → 时间通过beat切分对齐，声部通过位置明确区分
```

### 15.8 数学解释

**传统编码（如MIDI事件）建模的联合分布**：
$$P(X, Y) = P(e_1, e_2, ..., e_N)$$

其中事件 $e_i$ 可能属于Part0或Part1，模型需要隐式学习声部归属。

**本方法建模的条件分布**：
$$P(Y|X) = \prod_{t=1}^{T} P(y_t | x_{1:t}, y_{1:t-1})$$

通过显式的位置交错，声部归属由**序列结构决定**，而非模型预测。

## 十六、方法可行性的充要条件总结

本方法能够实现实时交错生成，需要同时满足以下条件：

### 16.1 模型架构要求

1. **因果注意力**：保证训练和推理的一致性
2. **自回归生成**：支持逐token流式输出
3. **KV缓存支持**：实现高效增量推理

### 16.2 编码方式要求

1. **声部边界清晰**：位置决定声部，而非预测声部
2. **时间对齐机制**：Beat级分割提供同步点
3. **独立结束标记**：明确每个声部每拍的边界
4. **序列长度可控**：稀疏编码压缩序列

### 16.3 训练策略要求

1. **选择性损失**：只对Part1计算损失
2. **交错数据构造**：输入序列符合[P0][P1][P0][P1]...模式
3. **时间偏移支持**：通过padding实现delay_beats

**三者缺一不可**：架构提供生成能力，编码提供结构约束，训练提供学习目标。
